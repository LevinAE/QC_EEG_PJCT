# Описание реализованного прототипа QC ЭЭГ

## Назначение проекта

Проект посвящён разработке минимального прототипа Quality Check для анализа качества ЭЭГ-записей. Основная идея состоит в том, чтобы не ограничиваться визуальным просмотром сигнала, а сформировать воспроизводимый программный конвейер, который рассчитывает количественные метрики качества, сохраняет результаты в машиночитаемом виде и создаёт человекочитаемый отчёт. В рамках текущей версии проект реализован без нейронных сетей и LLM, поэтому требования, связанные с `torchinfo` и Netron, к нему не применяются. Прототип относится к классу signal-processing решений и использует синтетические EEG-подобные данные для демонстрации запуска.

Репозиторий с работающим прототипом размещён на GitHub: https://github.com/LevinAE/QC_EEG_PJCT.

Выбор минимального варианта обусловлен практической задачей: необходимо получить работающий репозиторий, который можно запустить, показать промежуточные результаты и использовать как основу для дальнейшей интеграции с реальными BrainVision-записями. В приложенных материалах и литературном обзоре задача QC рассматривается как многоуровневая: проверка структуры файлов, оценка непрерывного сигнала, поиск проблемных каналов, анализ событий, эпох, ICA-компонент и формирование отчётов. Текущий MVP покрывает базовый уровень этой схемы: синтетическая запись, канальные амплитудные признаки, спектральные признаки, итоговый статус и отчётность.

## Реализованная функциональность

В проекте реализован CLI-запуск `python -m qc_eeg demo --output reports/demo_run`. Команда создаёт три демонстрационных записи. Первая запись имитирует нормальный EEG-подобный сигнал с выраженной альфа-активностью. Вторая запись содержит шумный канал `Fz`, в который добавлены широкополосный шум и компонент сетевой частоты. Третья запись содержит плоский низкоамплитудный канал `Pz`. Такой набор демонстрационных сценариев нужен для того, чтобы проверить разные типовые случаи QC: нормальная запись, высокоамплитудный/шумный канал и канал с аномально низкой амплитудой.

Модуль `qc_eeg.metrics` отвечает за генерацию данных и расчёт метрик. Для каждой записи вычисляются длительность, частота дискретизации, число каналов, средняя абсолютная амплитуда, максимальная амплитуда peak-to-peak, доминирующая частота, спектральная мощность по диапазонам Delta, Theta, Alpha, Beta и Gamma, отношение `alpha_beta_ratio` и отношение `theta_alpha_ratio`. На уровне каналов вычисляются `peak_to_peak_uv`, стандартное отклонение и средняя абсолютная амплитуда. Канал получает статус `BAD`, если его амплитуда или дисперсия выходят за заданные пороги либо если он близок к плоскому сигналу.

Модуль `qc_eeg.reporting` отвечает за экспорт результатов. Он создаёт две таблицы: `qc_results.csv` со сводными метриками по записям и `test_results.csv` с результатами демонстрационного тестирования. Также создаются два графика в формате SVG. Первый график показывает спектральную мощность по частотным диапазонам, второй показывает число проблемных каналов в каждой записи. Оба графика содержат название, подписи осей и легенду или цветовое обозначение статусов. Финальный HTML-отчёт `report.html` объединяет таблицы и графики в одном документе.

## Текущее состояние разработки

Текущий прототип является минимальной, но запускаемой версией. Он не требует закрытых EEG-файлов и может быть проверен на любой машине, где установлен Python и NumPy. Это важно для промежуточной сдачи, потому что преподаватель или научный руководитель сможет воспроизвести основные функции без доступа к лабораторному хранилищу данных. В репозитории предусмотрены автоматические тесты на базе `unittest`: они проверяют, что чистая запись получает статус `PASS`, шумная запись помечает канал `Fz`, плоская запись помечает канал `Pz`, а спектральные признаки являются числовыми.

При этом прототип пока не является полноценной системой обработки реальных ЭЭГ-записей. В исходных материалах уже есть код, рассчитанный на MNE, BrainVision-файлы, FASTER и HTML-шаблон отчёта. Однако этот код зависит от внешних конфигураций, локальных путей, библиотеки `eeg_auto_tools` и реальных данных. Поэтому в MVP эти зависимости сознательно не включены в обязательный путь запуска. Вместо этого сделана автономная демонстрационная версия, которая повторяет общую логику QC: входные записи, расчёт метрик, выявление проблемных каналов, сохранение таблиц и построение графиков.

## Таблицы и графики

В результате запуска формируются две таблицы. Таблица 1 содержит сводные QC-метрики по записям: идентификатор записи, сценарий, длительность, частоту дискретизации, число каналов, список плохих каналов, число плохих каналов, среднюю амплитуду, отношение alpha/beta и итоговый статус. Источник данных для этой таблицы - синтетические EEG-подобные записи, созданные внутри прототипа. Таблица 2 содержит результаты тестирования: название проверки, источник данных, ожидаемый результат, фактический результат и статус. Источник данных - демонстрационный запуск CLI.

График 1 показывает спектральную мощность по диапазонам Delta, Theta, Alpha, Beta и Gamma. Он нужен для проверки того, что прототип умеет формировать частотные признаки, которые часто используются при оценке качества EEG-сигнала. График 2 показывает количество проблемных каналов по записям. Он нужен для быстрой визуальной оценки результата QC и демонстрирует различие между чистой записью, записью с шумом и записью с плоским каналом.

## Анализ соответствия задачам

Промежуточные результаты соответствуют минимальному варианту поставленной задачи. Основные функции запускаются, результаты сохраняются в таблицы, графики формируются автоматически, HTML-отчёт создаётся без ручной сборки. Это закрывает демонстрационную часть практики: можно показать, что существует воспроизводимый конвейер, который принимает набор записей, считает метрики качества и выдаёт отчёт.

Что работает: генерация демонстрационных данных, амплитудный QC, спектральный QC, маркировка проблемных каналов, CSV-экспорт, HTML-отчёт, SVG-графики и автоматические тесты. Что требует доработки: чтение реальных BrainVision-файлов, использование монтажей `.elc`, интеграция с MNE Report, подключение FASTER через `mne-faster`, проверка событийной разметки, анализ эпох и ICA-компонент. Эти блоки уже методологически обоснованы в обзорах и частично присутствуют в исходном прототипе, но для стабильной сдачи MVP они вынесены в следующий этап.

## Дальнейшее развитие

Следующий технический шаг - добавить полноценный режим обработки реальных данных. Для этого нужно убрать жёсткие локальные пути из исходного скрипта, перенести параметры в JSON-конфиги, добавить команду вида `python -m qc_eeg run --data <path> --config <path>`, подключить MNE для чтения BrainVision и сохранить совместимость с демонстрационным режимом. После этого можно будет вернуть FASTER, MNE Report и расширенный HTML-шаблон. Отдельным этапом можно добавить event-level QC, epoch-level QC и ICA-level QC. Нейросетевой блок в текущей версии не планируется, чтобы не расширять требования к сдаче и не создавать необходимость готовить `torchinfo` и Netron.
